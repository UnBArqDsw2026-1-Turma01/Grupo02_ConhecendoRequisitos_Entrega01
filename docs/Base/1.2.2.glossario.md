# Glossário

## 1. Introdução

Um **glossário** é um instrumento de documentação que reúne, de forma ordenada e sistematizada, os termos e conceitos relevantes a um domínio específico, acompanhados de suas respectivas definições. Segundo o *Guia PMBOK* (Project Management Institute, 2021), o glossário constitui parte integrante do plano de gerenciamento das comunicações de um projeto, pois estabelece uma linguagem comum entre todos os envolvidos  (desenvolvedores, analistas, stakeholders e usuários), reduzindo ambiguidades e mal-entendidos ao longo do ciclo de vida do produto.

No contexto da Engenharia de Software, a IEEE Std 610.12-1990 (*IEEE Standard Glossary of Software Engineering Terminology*) define glossário como "uma lista de termos e suas definições", o reconhecendo como artefato essencial para garantir a precisão semântica na comunicação técnica. Wiegers e Beatty (2013), em *Software Requirements*, reforçam que a definição clara de vocabulário é um dos pilares da Engenharia de Requisitos de qualidade, pois evita que diferentes partes do sistema sejam concebidas com entendimentos divergentes sobre o mesmo conceito.

Este glossário cobre os domínios que constituem o projeto **ConhecendoRequisitos**: a plataforma em si (seus mecanismos, funcionalidades e componentes) e a área de **Engenharia de Requisitos** (campo de conhecimento ensinado pela plataforma).

### 2.1 Plataforma e Sistema

| Termo | Definição |
|---|---|
| **ConhecendoRequisitos** | Plataforma web de microlearning voltada ao ensino de Engenharia de Requisitos. Inspirada na dinâmica do Duolingo, a plataforma transforma o aprendizado técnico em uma experiência interativa, fragmentada em desafios de curta duração. |
| **Artefato de Aprendizado** | Qualquer produto gerado ou consumido no processo de ensino dentro da plataforma, como questões, trilhas, feedbacks e explicações técnicas associadas a erros cometidos pelo usuário. |
| **Constância** | Comportamento de estudar de forma regular e periódica. A plataforma incentiva a constância por meio das ofensivas, tornando o hábito de estudo diário recompensador. |
| **Aba do Aluno** | Painel de controle pessoal do usuário que exibe seu progresso nas trilhas, histórico de ofensivas e desempenho por categoria de requisito. |
| **Desafio** | Unidade mínima de interação na plataforma. Consiste em uma atividade, geralmente um quiz em que o usuário deve completar em até 5 minutos. |
| **Explicação de Erro** | Conteúdo didático exibido imediatamente após o usuário responder uma pergunta em um desafio. Tem como objetivo reforçar o conceito técnico no exato momento da dúvida, aproveitando o estado cognitivo de atenção elevada. |
| **Engajamento** | Grau de envolvimento ativo do usuário com a plataforma, medido por métricas como frequência de acesso, tempo por sessão e taxa de conclusão de desafios. |
| **Feedback Imediato** | Mecanismo pelo qual a plataforma responde instantaneamente à ação do usuário — correta ou incorreta, fornecendo reforço positivo ou explicação corretiva É o núcleo pedagógico da plataforma. |
| **Jogo de Classificação** | Modalidade central de desafio da plataforma, na qual o usuário deve categorizar, associar ou classificar elementos relacionados à Engenharia de Requisitos (ex.: identificar o tipo de um requisito, classificar uma técnica de elicitação, etc.). |
| **Microlearning** | Abordagem pedagógica que fragmenta o conteúdo em pequenas unidades de aprendizado autossuficientes, consumíveis em poucos minutos. Na plataforma, cada desafio representa uma unidade de microlearning. |
| **Módulo** | Agrupamento de desafios que cobrem um subtópico específico dentro de uma trilha. Ex.: dentro da trilha de Elicitação, pode existir um módulo dedicado a Entrevistas, Questionários, etc. |
| **Ofensiva** | Contador que registra quantos dias consecutivos o usuário realizou ao menos um desafio. É o principal mecanismo de incentivo à constância e regularidade nos estudos. |
| **Plataforma Web** | Sistema acessado via navegador de internet, sem necessidade de instalação local. O ConhecendoRequisitos é projetado como site para garantir acessibilidade multiplataforma (desktop, tablet e smartphone). |
| **Quiz** | Formato de avaliação baseado em perguntas e alternativas de resposta. Na plataforma, os quizzes são o principal veículo dos desafios, podendo ser de múltipla escolha, verdadeiro/falso ou classificação. |
| **Sistema de Progressão** | Conjunto de regras que determina como o usuário avança entre módulos e trilhas, geralmente baseado em desempenho mínimo (acertos)|
| **Trilha Especializada** | Sequência estruturada de módulos e desafios organizada em torno de uma etapa ou competência da Engenharia de Requisitos (ex.: Elicitação, Análise, Modelagem, Validação). Permite ao aluno focar nas áreas de maior dificuldade. |

### 2.2 Engenharia de Requisitos (ER)

| Termo | Definição |
|---|---|
| **Análise de Requisitos** | Etapa da ER dedicada a examinar os requisitos elicitados, identificar conflitos, ambiguidades, redundâncias e lacunas, e refinar o entendimento do que o sistema deve fazer. É uma das trilhas especializadas da plataforma. |
| **Caso de Uso (Use Case)** | Técnica de modelagem que descreve as interações entre um ator (usuário ou sistema externo) e o sistema de software para atingir um objetivo. Faz parte do conteúdo da trilha de Modelagem. |
| **Critério de Aceitação** | Condição que um requisito deve satisfazer para ser considerado implementado corretamente. Define o "pronto" de cada funcionalidade do ponto de vista do stakeholder. |
| **Documento de Requisitos** | Artefato formal que registra os requisitos de um sistema de software de maneira estruturada. Pode seguir modelos como o IEEE 830 (SRS — Software Requirements Specification). |
| **Elicitação de Requisitos** | Processo de descoberta e coleta dos requisitos junto aos stakeholders, usando técnicas como entrevistas, questionários, observação, workshop, prototipação e análise de documentos. É a primeira trilha especializada da plataforma. |
| **Engenharia de Requisitos (ER)** | Subárea da Engenharia de Software dedicada à identificação, análise, especificação, validação e gerenciamento dos requisitos de um sistema de software. Segundo Sommerville (2019), ER é o processo de estabelecer os serviços que o cliente requer do sistema e as restrições sob as quais ele deve operar. |
| **Engenharia de Software** | Disciplina que aplica princípios de engenharia ao desenvolvimento, operação e manutenção de software. A ER é uma de suas subáreas fundamentais (Pressman & Maxim, 2021). |
| **Especificação de Requisitos** | Atividade de documentar os requisitos levantados em um formato preciso, claro e verificável. O resultado é o Documento de Requisitos (SRS). |
| **Gerenciamento de Requisitos** | Conjunto de atividades para controlar mudanças nos requisitos ao longo do ciclo de vida do projeto, mantendo rastreabilidade e consistência. |
| **Histórias de Usuário (User Stories)** | Formato ágil de especificação de requisitos que descreve uma funcionalidade da perspectiva do usuário: "Como [papel], quero [funcionalidade] para [benefício]." |
| **Modelagem de Requisitos** | Representação gráfica ou formal dos requisitos para facilitar sua compreensão e comunicação. Inclui diagramas UML, casos de uso, modelos de domínio, entre outros. É uma das trilhas especializadas da plataforma. |
| **Priorização de Requisitos** | Processo de ordenar os requisitos por relevância, valor de negócio, risco ou esforço, subsidiando decisões sobre o que entregar primeiro. Técnicas comuns incluem MoSCoW e Kano. |
| **Prototipação** | Técnica de elicitação e validação que consiste em criar versões preliminares do sistema (protótipos) para obter feedback antecipado dos stakeholders. |
| **Rastreabilidade** | Capacidade de relacionar cada requisito à sua origem (stakeholder, documento, necessidade de negócio) e aos artefatos de design, código e teste derivados dele. |
| **Requisito** | Declaração do que um sistema deve fazer (funcional) ou como deve se comportar (não funcional). Segundo a IEEE 610.12-1990, é "uma condição ou capacidade necessária a um usuário para resolver um problema ou atingir um objetivo". |
| **Requisito Funcional (RF)** | Requisito que descreve um comportamento ou função específica que o sistema deve executar. Ex.: "O sistema deve permitir que o usuário se cadastre com e-mail e senha." |
| **Requisito Não Funcional (RNF)** | Requisito que descreve uma restrição de qualidade ou atributo do sistema (desempenho, segurança, usabilidade, confiabilidade). Ex.: "O sistema deve responder em menos de 2 segundos para 95% das requisições." |
| **Stakeholder** | Qualquer pessoa ou grupo com interesse legítimo no sistema de software, podendo ser afetado por ele ou exercer influência sobre seu desenvolvimento. Inclui clientes, usuários finais, desenvolvedores, gerentes e reguladores. |
| **Técnica de Elicitação** | Método utilizado para extrair requisitos dos stakeholders. Exemplos: entrevista estruturada, grupo focal, observação etnográfica, análise de documentos, brainstorming e JAD (*Joint Application Development*). |
| **Validação de Requisitos** | Etapa que verifica se os requisitos documentados refletem corretamente as necessidades dos stakeholders e se são consistentes, completos e verificáveis. É uma das trilhas especializadas da plataforma. |
| **Verificação de Requisitos** | Processo que confirma se os requisitos estão corretos, sem ambiguidades e tecnicamente viáveis, antes do início do desenvolvimento. |

### 2.3 Tecnologia e Desenvolvimento Web

| Termo | Definição |
|---|---|
| **API (Application Programming Interface)** | Interface que permite a comunicação entre sistemas. A plataforma pode utilizar APIs para integração com serviços externos (autenticação, envio de e-mails, analytics, etc.). |
| **Back-end** | Camada do sistema responsável pela lógica de negócio, persistência de dados e regras de acesso. No ConhecendoRequisitos, o back-end gerencia usuários, trilhas e progresso. |
| **Banco de Dados** | Sistema de armazenamento persistente dos dados da plataforma, incluindo usuários, questões, respostas, trilhas e ofensivas. |
| **Front-end** | Camada do sistema com a qual o usuário interage diretamente, responsável pela interface visual e experiência de uso. O front-end da plataforma implementa os princípios de microlearning. |
| **Plataforma Web** | Ver definição na seção 2.1. |
| **Responsividade** | Capacidade da interface de adaptar seu layout a diferentes tamanhos de tela (computador, tablet, smartphone), essencial para suportar o uso em deslocamento. |
| **SPA (Single Page Application)** | Arquitetura de aplicação web em que o conteúdo é carregado dinamicamente sem recarregamento completo da página, proporcionando experiência mais fluida,  característica desejável em plataformas de microlearning. |
| **UX/UI Design** | Conjunto de práticas de design voltadas à experiência do usuário (UX) e à interface visual (UI). No ConhecendoRequisitos, o design é inspirado no Duolingo, priorizando simplicidade, cor e recompensa visual. |
