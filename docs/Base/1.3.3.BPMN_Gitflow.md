# BPMN: Gitflow

## Introdução

O **Gitflow** é um modelo de ramificação (_branching model_) para o Git, proposto por Vincent Driessen em 2010, amplamente adotado em projetos de desenvolvimento de software por organizar de forma clara e previsível o ciclo de vida do código.

Seu principal objetivo é separar o trabalho em andamento do código estável, definindo branches com papéis bem determinados: `main`, `develop`, `feature` e `release`. Essa separação permite que múltiplos desenvolvedores colaborem em paralelo sem interferir uns nos outros, mantendo a branch principal sempre em estado de produção.

Durante as próximas etapas de desenvolvimento deste projeto, o Gitflow será utilizado como fluxo padrão de trabalho. Toda nova funcionalidade partirá de uma branch `feature` criada a partir da `develop`; ao ser concluída e revisada, será integrada de volta à `develop`. Quando o conjunto de funcionalidades estiver estável e pronto para entrega, uma branch `release` será gerada para os ajustes finais e, após aprovação, integrada à `main` com a devida tag de versão. ## Descrição do fluxo

### 1. Desenvolvedor

O processo se inicia com o desenvolvedor criando uma branch `feature` a partir da `develop`. Em seguida, o código é implementado localmente e os testes são executados. Caso os testes falhem, o código é corrigido e os testes são re-executados. Quando os testes passam, as alterações são enviadas ao repositório remoto e um **Pull Request (PR)** é aberto para a branch `develop`.

Caso o revisor solicite ajustes, o desenvolvedor aplica as correções e atualiza o PR. O ciclo se repete até que o código seja aprovado.

### 2. Revisor

Ao receber a notificação do PR, o revisor analisa o código. O gateway de decisão **"Aprovado no review?"** determina o caminho seguinte:

- **Não aprovado:** o revisor solicita ajustes por meio de comentários no PR, e o fluxo retorna ao desenvolvedor.
- **Aprovado:** o merge para a branch `develop` é realizado.

Após o merge, um segundo gateway verifica se o estado atual da `develop` representa uma **versão estável para release**:

- **Sim:** uma release é gerada, disparando o processo na raia `main`.
- **Não:** o fluxo retorna ao início, aguardando novas features.

### 3. main

Ao receber a release gerada pelo revisor, a raia `main` executa o merge do código na branch principal e, em seguida, realiza o **deploy** (publicação do site/aplicação). O processo encerra com o sistema em produção atualizado.

## Elementos BPMN utilizados

A Tabela 2 destaca os elementos BPMN que foram utilizados.

Tabela 2: Elementos BPMN utilizados

| Elemento | Tipo | Descrição |
|----------|------|-----------|
| Círculo simples | Evento de início | Marca o ponto de partida do processo |
| Círculo com borda dupla | Evento de fim | Indica o encerramento do processo |
| Círculo com seta | Evento intermediário de link | Conecta fluxos distantes sem cruzar setas longas |
| Retângulo arredondado | Tarefa (_Task_) | Representa uma atividade executada por um participante |
| Losango com X | Gateway exclusivo | Ponto de decisão: apenas um caminho é seguido <br> Ponto de convergência: independente de qual chega nele, a próxima task é obrigatória|
| Seta tracejada | Fluxo de mensagem | Comunicação entre participantes (pools) diferentes |
| Seta contínua | Fluxo de sequência | Ordem de execução dentro de um mesmo participante |
